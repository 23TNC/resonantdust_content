; Game conventions that the DSL paths encode. SYNTAX.txt covers tokens; this
; covers what the tokens MEAN, so the editor (and us) can highlight by role.

==============================================================================
STACKS & ROOTS
==============================================================================

Every chain hangs off one root card. A root currently exposes three stacks:

    stack 0  HEX     the tile / hex slot under the root  (STACK_DIRECTION HEX)
    stack 1  TOP     grows upward from the root          (STACK_DIRECTION UP)
    stack 2  BOTTOM  grows downward from the root        (STACK_DIRECTION DOWN)

So the first number after `slot` is the STACK selector (0|1|2), not a free
index. The root itself may be promoted into a stack at offset 0 — a recipe
reference to `slot.<stack>.0` can resolve to the root card.

==============================================================================
THE SLIDING FRAME  —  slot.a.b
==============================================================================

`slot` is the recipe's entry point: a recipe pattern is written against
`slot.<stack>.<offset>` coordinates, and the matcher slides that frame along
each stack looking for a window where the pattern holds.

    slot.a.b   ==   root[a][b + N]   for some N in 0 .. len(root[a])

  - a = stack       (0 hex / 1 top / 2 bottom)
  - b = offset      position WITHIN the recipe's frame, not an absolute index
  - N = frame base  chosen by the matcher as it slides the window

This is why `slot` is special: it is NOT root and NOT an absolute address. It
is a sliding offset from root. `slot.1.0` = "top stack, first card of the
current frame"; the same recipe matches anywhere the frame can sit.

`root` (as a path anchor) refers to the chain root directly — no sliding.

==============================================================================
PATH ANATOMY
==============================================================================

A path is dot-separated segments. Segments fall into distinct ROLES:

  anchor       leading segment that picks the address space:
                 slot   sliding recipe frame (followed by a stack.offset pair)
                 root   the chain anchor, absolute
                 var    engine scratch register     (var.<n>)
                 sys    engine-provided field        (sys.duration)
                 aspect aspect namespace of the current card
                 objects / color / asset / shape / biome  card-instance data

  coord pair   the `<stack>.<offset>` immediately after `slot`
                 stack  = 0|1|2 (constrained enum: hex/top/bottom)
                 offset = frame-relative index

  hop          a segment that re-anchors the path onto ANOTHER card:
                 owner  walk to the card that owns the current one, then the
                        rest of the path is resolved in THAT card's space
                        (e.g. slot.1.0.owner.slot.1.0 — actor's own top stack)
                 inventory / blueprint  container destinations for `create`

  field        a named leaf that reads a property, not a child collection:
                 def_id  the card's definition key
                 style   render direction / style
                 (and aspect names: aspect.wood, aspect.corpus+, ...)

  index        a numeric segment that is an array/collection index rather than
               a stack coordinate: objects.0, var.0, asset.<n>

  interp       a value reference embedded inside a path as a dynamic index:
                 objects.*var.0   →  objects[ value of var.0 ]
                 aspect.*var.2    →  aspect indexed by var.2

==============================================================================
ASPECTS
==============================================================================

The trailing non-numeric segment of an aspect path is the aspect NAME
(slot.0.0.aspect.stone → stone). Satisfaction is a `satisfies` LUT, not a tree
(content/data/aspect/01.rd): each aspect lists the aspects it satisfies, so
`pine → [wood]` (and an aspect can satisfy several). A predicate `aspect.Q`
folds over a card: sum every stocked aspect A where A==Q or Q ∈ satisfies(A) —
so a tile with `pine:2` + `ash:3` reads `aspect.wood` = 5. Exact match is
implicit (no aspect lists itself). The gate computes the transitive closure +
sum and writes the rolled-up key into the operating-set frame; the VM reads it
as a plain key. Polarity suffixes `+` / `-` are part of the name (corpus+).

==============================================================================
VISUAL PRIMITIVES  (:visuals -> &prims)
==============================================================================

`:visuals` builds the `prims` draw list, but the DSL never hand-builds it: the
ENGINE owns `prims`. You construct a primitive with a `^` intrinsic, which
appends it to `prims` and hands back a HANDLE; you configure the prim through the
handle. The client reconciles `prims` into retained Pixi objects (by index) and
eases each toward the values you set. The DSL only ever writes TARGET values; the
engine owns the live objects + the tween, so you never hold a sprite -- you
construct one and describe it.

    ^hex call   &h set        ; engine pushes a hex onto prims, returns a handle
    0.0 0.0     &h.pos  vec2   ; &h.<field> writes THROUGH to that prim
    100.0 100.0 &h.size vec2
    *color.bg   &h.tint set

`&h` is a reference to the prim just pushed; reuse the name for the next one
(`^sprite call &h set`). Push order IS paint/reconcile order -- no index
bookkeeping. (`prims` exists because `^` filled it; the engine never scans for a
magic variable, and you never write `&prims.0.kind`.)

Kinds are `^` engine intrinsics (NOT a `$` registry): ^rect | ^hex | ^text |
^sprite. The engine (`vm::PRIM_KINDS`) + the client `makePrimitive` own the
matching set.
Fields (all optional; the kind is fixed by the `^` constructor):
  pos size anchor vec2, CARD-SPACE 0..100 (0 = left/top edge, 100 = right/bottom;
                  anchor is the 0..100 pivot). The client scales by the card's
                  pixel box x dpr -- so one spec is correct at any size / density.
  scale rot alpha floats (defaults 1 / 0 / 1).
  tint            #rrggbb (for a rect/hex fill this IS the colour: white texture
                  x tint).
  texture         a bare asset / render-object NAME (rect/hex fill image, or the
                  sprite art). The CLIENT picks the LOD variant + faction -- the
                  definition just names the thing.
  text            a LOCALE KEY. The client resolves it via the locale runtime;
                  definitions never decode locales (locales = key->string).

Hooks: build the prims in `@init` (first draw -- the client snaps current=target)
and/or `@update` (a change -- the client eases to the new targets). Both run the
same builder; the VM emits the full list each time and the client diffs by index.
Conditional elements toggle via `alpha 0`, never by adding/removing a record (the
index is the identity). Genuine per-frame motion is a dedicated engine primitive,
not DSL.

Shared builders compose the standard sets: e.g. `ring_prims` for tiles emits the
hex body (tint `color.bg`) plus one sprite per stock object on a trig ring, the
sprite's `texture` being the aspect's render object (`*rec.art.object`). World
tiles render through `tile_prims`, which overlays the STORED stock (not biome-
recomputed) before running `:visuals`.

==============================================================================
HIGHLIGHTING IMPLICATIONS
==============================================================================

The editor highlights references with a deliberately simple two-part split,
rather than coloring every role above (which read as noise):

  HEAD  the sigil + path up to the last `.` (or to whitespace if no dot),
        colored by the SIGIL so $ / & / * reads are distinguishable at a glance.
  TAIL  the segment after the last `.` — the aspect / asset / variable being
        edited — in its own color. A dotless path, or one whose final segment
        is a number (slot.1.0), is all head.

The roles above (anchor, coord pair, hop, interp) still describe what the head
MEANS; they are just not individually colored.
