# data/

Shared game data ‚ÄĒ JSON files read by **both** the SpacetimeDB server (Rust)
and the pixijs client (TypeScript). Anything that defines what cards are,
which recipes exist, or what aspects mean lives here.

## Build pipeline

Two consumers, two delivery mechanisms ‚ÄĒ both view the same files:

- **Server**: [`spacetime/compose.yml`](../spacetime/compose.yml) mounts
  `../data` (this directory) at `/workspace/server/spacetimedb/data` inside
  the build container. The Rust module embeds files at compile time via
  `include_str!("../data/...")` ‚ÄĒ see
  [`spacetime/server/spacetimedb/src/definitions.rs`](../spacetime/server/spacetimedb/src/definitions.rs).
  For host-side `cargo` builds (outside the container), create a symlink at
  `spacetime/server/spacetimedb/data` ‚Üí `../../../data` so the same paths
  resolve.
- **Client**: this directory is symlinked into the pixijs project so the
  client reads the same JSON at runtime.

A change to any file here ships to both halves. They MUST agree on schema.

## Layout

Files are grouped by topic. Each topic directory holds its registries and
metadata at the top, with bulky definition files in a `data/` subdirectory:

```
data/
  cards/
    types.json     # card_type / card_category registry
    aspects.json   # aspect catalog
    flags.json     # bit-position registry for cards.flags
    id.json        # stable definition_id (generated)
    data/*.json    # card definition buckets
  recipes/
    types.json     # recipe_type / recipe_category registry
    flags.json     # bit-position registry for actions.flags
    id.json        # stable recipe id (generated)
    data/*.json    # recipe definition buckets
  zones/
    biomes.json    # biome registry for procedural map gen
  bootstrap/
    bootstrap.json # dev-only seed data
  gen-ids.py       # ID generator
```

## Files

| File | Status | Purpose |
| --- | --- | --- |
| [`cards/types.json`](cards/types.json) | live | `card_type` and `card_category` id registry. Authoritative source for the type-name ‚Üí id mapping baked into `packed_definition`. Visibility / shape rules live in `_rules`. |
| [`cards/aspects.json`](cards/aspects.json) | live | Aspect catalog ‚ÄĒ the tags placed on cards (`labor`, `combat`, `fleeting`, ‚Ä¶) grouped by domain. Server assigns 1-indexed `AspectId`s in JSON insertion order across groups. |
| [`cards/flags.json`](cards/flags.json) | live | Bit-position registry for the `cards.flags` u8 column. Each entry pins an explicit `bit` in `[0, 7]`. Append-only: removed flags leave `_reserved_<bit>` tombstones. Currently *documentation only* ‚ÄĒ the server doesn't load this file yet. |
| [`cards/id.json`](cards/id.json) | live, generated | Stable `definition_id` for every card, scoped per `(card_type, category)`. Assigned once and never reassigned. Generated by `gen-ids.py`. |
| [`cards/data/*.json`](cards/data/) | live | Card definitions. See [`cards/AGENTS.md`](cards/AGENTS.md). |
| [`recipes/types.json`](recipes/types.json) | live | `recipe_type` / `recipe_category` id registry, used by `pack_recipe`. |
| [`recipes/flags.json`](recipes/flags.json) | live | Bit-position registry for the `actions.flags` u8 column. Same conventions as `cards/flags.json`. Documentation only today. |
| [`recipes/id.json`](recipes/id.json) | live, generated | Stable integer ID for every recipe (what `Action.recipe` stores on the wire). Generated by `gen-ids.py`. |
| [`recipes/data/*.json`](recipes/data/) | live | Recipe definitions. See [`recipes/AGENTS.md`](recipes/AGENTS.md). |
| [`zones/biomes.json`](zones/biomes.json) | live | Biome registry for procedural map generation. Each biome has a center in climate space (`temperature`, `humidity` ‚ąą `[0, 1]`) and a weighted tile distribution (keys resolved through `cards/id.json` under `tile/default/<key>`). The map generator samples a climate vector per cell, weights biomes by inverse-square distance from the cell's climate point, blends their tile distributions, and weighted-picks a `definition_id`. Biomes are category-agnostic ‚ÄĒ the zone's category byte selects the visual variant. |
| [`gen-ids.py`](gen-ids.py) | tooling | Generates `cards/id.json` and `recipes/id.json` by globbing `cards/data/*.json` and `recipes/data/*.json`. **Run before `spacetime build` whenever you add or remove cards or recipes** ‚ÄĒ the server's registry build refuses to load entries missing from these files. IDs are append-only; removed entries leave tombstones so existing IDs never get recycled. |
| [`bootstrap/bootstrap.json`](bootstrap/bootstrap.json) | live | Seed data loaded by the dev-only `bootstrap` reducer in [`spacetime/server/spacetimedb/src/debug.rs`](../spacetime/server/spacetimedb/src/debug.rs). Sections: `player` (find-or-create, idempotent on `name_normalized`), `zones` (replace-on-insert; `(zone_q, zone_r)` packs into `Zone.macro_zone`, `(card_type, category)` into the shared nibble byte, `t0..t7` go into the eight `u64` cell rows), `card` (additive; each entry resolves through `cards/id.json` and lands in the first player's inventory). Per-zone `layer` is parsed but not stored ‚ÄĒ the Zone schema today identifies a chunk by `macro_zone` alone. |

## Conventions across all files

- **Valid JSON, no comments.** JSON-with-comments is not valid JSON. Use
  `_comment` keys (or `_rules`, `_notes`, etc.) for inline documentation
  ‚ÄĒ both the server's `json_id_map` parser and the recipe parser skip
  any key beginning with `_`.
- **Type and category names are singular** (`"discipline"`, not
  `"disciplines"`). The bucket-name field in `cards/data/*.json` MUST match
  the key in `cards/types.json`'s `types` map exactly (case-sensitive). A
  silent mismatch causes the bucket to be skipped at registry build with
  no error ‚ÄĒ you'll only notice when a card "fails to decode" at runtime.
- **Reserved keys**: anything beginning with `_` is parser-skipped.
  `_reserved_*` slots in `cards/types.json` reserve numeric ids for future
  types; they don't register as real types and bucket names like
  `"_reserved_1"` will never resolve.
- **1-based external ids, 0 reserved as sentinel.** `definition_id` and
  `AspectId` are both 1-indexed; a 0 in either context means "no
  card / no aspect". Don't reorder JSON object keys casually ‚ÄĒ the
  position determines the id.
- **Adding a new file requires updating the loader.** The server can't
  glob `include_str!`; each new `cards/data/NN.json` or `recipes/data/NN.json`
  needs a corresponding tuple appended to `CARDS_FILES` /
  `RECIPES_FILES` in [`definitions.rs`](../spacetime/server/spacetimedb/src/definitions.rs).
  Forgetting this is a "my file isn't loading" first-suspect.
- **Adding cards or recipes requires running `gen-ids.py`.** The
  registry rejects entries that aren't in `cards/id.json` /
  `recipes/id.json` with a clear "run gen-ids.py" error message.
  Re-running is idempotent ‚ÄĒ it only assigns IDs to new entries.
- **Reserved card keys.** Card keys cannot be `"any"` (the recipe-side
  wildcard) and cannot start with `@` (the recipe-side card-type
  prefix). The current data trips neither.

## Failure modes

- **Server**: a malformed file fails the registry build. The server's
  `OnceLock<Result<‚Ä¶, String>>` cells store the error and every lookup
  returns it ‚ÄĒ the module doesn't crash, but recipes/cards just won't
  resolve. Check the server logs / first-call error message.
- **Client**: TBD ‚ÄĒ depends on the pixijs loader's error handling.
- **Cross-side schema drift**: nothing enforces server and client agree
  on what they're reading. If you add a field that the server uses but
  the client doesn't (or vice versa), you'll find out at integration
  time. Worth a thought before adding fields.

## Where parsing lives

- Server: [`spacetime/server/spacetimedb/src/definitions.rs`](../spacetime/server/spacetimedb/src/definitions.rs)
  ‚ÄĒ `build_aspects`, `build_cards`, `build_recipes`, `parse_entity`,
  `parse_duration`, `json_id_map`. All `Result`-returning; failures
  propagate via the registry's stored error.
- Client: pixijs project (location varies ‚ÄĒ check the data/ symlink in
  pixijs).
