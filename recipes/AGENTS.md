# content/recipes/

Recipe registry. Under the [recipe-tape model](../../docs/RECIPE_TAPE_REWRITE.md),
every recipe is a flat key→`{ input, output }` object; statements are
path-prefixed strings parsed into a `RecipeDef` IR consumed by both the
SpacetimeDB shard module (Rust) and the PixiJS client (wasm via the
content crate).

The client matches; the server verifies. Recipe matching no longer runs
server-side — the client pre-resolves every `slot.<branch>.<index>`
reference and submits `(recipe_id, root, bindings: Vec<Vec<u32>>)`.

## Layout

```
content/recipes/
  AGENTS.md            ← this file
  id.json              ← stable u16 id per recipe key (generated by gen-ids.py)
  types.json           ← recipe_type / recipe_category registry
  data/*.json          ← recipe definitions. Auto-discovered at build.
```

## File format

Each file under `data/` is **flat** — top-level keys are recipe keys,
values are the recipe spec. (The legacy outer `stack.up` / `magnetic.up`
/ `on_create` buckets are gone; the recipe-tape parser inspects each
recipe's input/output statements to classify it.)

```jsonc
{
  "triple_corpus": {
    "input": [
      "use.slot.1.0.def_id: corpus",
      "claim.slot.1.1.def_id: corpus",
      "claim.slot.1.2.def_id: corpus"
    ],
    "output": [
      "sys.duration.set: 10",
      "slot.1.0.style.set: rtl",
      "slot.1.0.destroy"
    ]
  }
}
```

Multiple recipes per file are fine; files are split by topic.
Unknown top-level keys (e.g. `_comment` JSON-doc) are silently
skipped so content can outpace the parser.

Authority on the grammar lives in
[`content/src/recipe_tape.rs`](../src/recipe_tape.rs).

## Statement grammar

```
statement   := path ('.' op)? (': ' value)?
path        := segment ('.' segment)*
segment     := identifier | integer
identifier  := [a-z_][a-z0-9_+-]*
integer     := decimal digits
value       := string | integer
```

The last path segment usually IS the op (`def_id`, `aspect.X.min`,
`destroy`, `set`, `add`, `sub`, …). The grammar extends by adding
vocabulary (reserved tokens), not by adding enum variants — the
parser uses a closed reserved-word set; any segment outside it
that isn't a content-namespace identifier (aspect name, card key,
flag name) is a hard build error.

A single `': '` (colon + space) is the only legal whitespace inside
a statement.

## Slot accessor

Cards are referenced via a single uniform accessor:

```
slot.<branch>.<index>
```

- **Branch 0 (`slot.0.X`)** — the tile branch (visually beneath root;
  what was "hex" under the legacy model). Branch-0 supports a
  synthetic-tile sentinel: if the client passes `bindings[iter_0][0] == 0`,
  the server promotes the underlying zone tile to a card via
  `cards::find_or_create_tile_card` before evaluating predicates.
- **Branch 1 (`slot.1.X`)** — top stack ("up").
- **Branch 2 (`slot.2.X`)** — bottom stack ("down").
- **Branch 3+** — reserved.

`root` is sugar for "the recipe's root card" (a top-level binding that
isn't part of any branch chain). Index counts from 0.

## Input statements

Each input statement is a predicate evaluated as a conjunction at
match time (client) and verify time (server). The first path segment
declares the **iterator kind**:

| Kind | Hold acquired | Use |
| --- | --- | --- |
| `use.<path>.<predicate>` | none | "is this card present at this position?" — match-only, no hold |
| `claim.<path>.<predicate>` | `slot_hold_count` (exclusive) | recipe consumes / transforms this card |
| `borrow.<path>.<predicate>` | `slot_share_count` (shared) | recipe reads but doesn't consume; multiple concurrent borrows allowed |
| `share.<path>.<predicate>` | `slot_share_count` (shared) | alias of `borrow` |

After the iterator kind comes the path (a `slot.<branch>.<index>` or
`root`, optionally with `.owner` / `.parent` chain steps to traverse
into another card's branches), then the predicate:

```
<kind>.<path>.def_id: <card_key>            # bare-card predicate (equality)
<kind>.<path>.aspect.<name>.min: <int>      # aspect predicate (≥ min)
<kind>.<path>.flag.<flag_name>              # flag-presence predicate
```

`.aspect.<name>.min` requires `definitions::aspect_id_by_name(name)`
(the wasm `aspectIdByName` export) — predicates against undeclared
aspects fail at registry build.

### Nested iterators

A `slot.<branch>.<index>` segment can appear **mid-path** to reach into
another card's branch. The classic example is the `cut_tree` axe
predicate:

```
"claim.slot.1.0.owner.slot.1.0.def_id: axe"
```

Read this as: "the card at the actor's (`slot.1.0`'s) owner's top
branch slot 0 must be an axe." `chain_stitch` doesn't touch
nested-iterator bindings — their cards already live in other chains
(equipment, queued actions, …) and are predicate-verified but not
stitched.

## Output statements

The output tape is walked sequentially at completion. Statements
dispatch on the **last path segment**:

| Last segment | Effect |
| --- | --- |
| `sys.duration.set: <secs>` | Set the recipe's duration in seconds. Completion stamps `valid_at_time = start + secs * 1000`. |
| `sys.style.set: <style>` | Set the progress-bar style (`ltr` / `rtl` / `none`). Written to the actor's `flags_state.progress_style` bits. |
| `<path>.style.set: <style>` | Per-card style override (lands on that card's row). |
| `var.<N>.set: <value>` | Variable arithmetic (per-action scratch space; 8 `i32` slots). `<value>` is either an integer literal or a path expression read via `read_path_value`. |
| `var.<N>.add: <value>` / `var.<N>.sub: <value>` | Same; additive/subtractive. |
| `var.<N>.random: <max>` | RNG roll `0..max` into var `N`. **Pending** (Phase 12). |
| `when.<var-pred>.<inner>` | Predicate gate — recurses into `<inner>` only if the var predicate holds (`gt`, `ge`, `lt`, `le`, `eq`, `ne`). |
| `<path>.destroy` | Flip `FLAG_DEAD` on the resolved card at `completion_ms`. |
| `<path>.<chain>.inventory.create: <def_key>` | Spawn one card at the resolved location's inventory. Multi-spawn by repeating the statement. |
| `<path>.aspect.<name>.sub: <int>` / `.add` / `.set` | Stock write — decrements / increments / sets the row's stock slot for `<name>` on the resolved card. Tile-cards: stored in `flags_bk.tile_stock_{0,1}` and written via `cards::set_tile_stock`. Saturates at `[0, slot.max]`. |

### Output card refs

Output paths resolve the same way input paths do — `root`,
`slot.<branch>.<index>`, optionally with `.owner` / `.parent` chain
steps. `slot.0.0.owner.inventory.create: log` spawns a `log` in the
inventory container of the tile branch's slot 0.

## `id.json` and `types.json`

`id.json` is generated by `gen-ids.py` and assigns a stable u16 id per
recipe key. `types.json` declares the `recipe_type` / `recipe_category`
registry — under the recipe-tape model the active type is `Stack`
(directions `Up` / `Down` / `Hex`); legacy types (`OnCreate`,
`Magnetic`) are tombstoned. `Action.recipe` on the wire is
`pack_recipe(type_id, category_id, recipe_id)`.

Per-recipe-type and per-category IDs are 1-indexed; 0 is the
sentinel.

## Pitfalls

- **Reserved words are closed.** Aspect / card-key / flag names that
  shadow grammar tokens (`slot`, `root`, `aspect`, `min`, `destroy`,
  `create`, `set`, `add`, `sub`, `random`, `when`, `gt` / `ge` / `lt`
  / `le` / `eq` / `ne`, `owner`, `parent`, `inventory`, `def_id`,
  `flag`, `duration`, `style`, `var`, `sys`) fail at build. The
  canonical check lives in [`content/src/recipe_tape.rs`](../src/recipe_tape.rs).
- **`slot.<branch>.<index>` indexes from 0, not 1.** Sub-chain
  position references use 0-based offsets within the branch.
- **Predicate / consume coupling on stock.** If a recipe declares
  `slot.0.0.aspect.<name>.min: M` AND its output sub-trees that
  aspect (`slot.0.0.aspect.<name>.sub: N`), the parser enforces
  `M >= N` so a match can't fire that would underflow the stock.
- **Recipe ids are globally unique across files** — two files
  can't define a `triple_corpus`. Build errors with the conflicting
  files named.
- **Nested-iterator bindings aren't chain-stitched.** Only top-level
  iterator bindings (parent path empty) get `OnRoot` / `Slot`
  stamping at propose time. Cards reached via a mid-path
  `slot.<branch>.<index>` stay in whatever chain they were in.

## Adding a new recipe

1. Pick (or create) a file under [`data/`](data/) — files are
   organisational, the parser walks all of them.
2. Add a top-level entry: `"<recipe_key>": { "input": [...], "output": [...] }`.
3. Write input statements covering every binding the recipe needs.
   Bare-card predicates use `def_id`; aspect-bound predicates use
   `aspect.<name>.min`.
4. Write the output tape — durations, destroys, creates, stock
   writes, in execution order.
5. Run `bin/content check`. `gen-ids.py` assigns the recipe id;
   the registry build catches grammar / reserved-word / predicate
   coverage issues.
6. Add a label / success message to
   [`../locales/recipes/en.json`](../locales/recipes/) under the
   recipe key. Without a locale entry, clients display the bare
   recipe key.

## Where parsing lives

- [`content/src/recipe_tape.rs`](../src/recipe_tape.rs) — the parser
  + IR. `parse_file` ingests a JSON file, `parse_statement` is the
  single-statement parser, `resolve_slots` extracts iterators and
  their parent paths.
- [`content/src/recipe_core.rs`](../src/recipe_core.rs) — registry
  + lookups. Thin shim around `recipe_tape::parse_file`:
  `recipe(id)`, `find_recipe(key)`, `recipes_by_priority()`.
- Client matcher:
  [`pixijs/src/game/actions/recipeMatcher.ts`](../../pixijs/src/game/actions/recipeMatcher.ts).
  Walks the recipe catalog in `AnchorSet.priorityKey` order,
  resolves slots via the input branch chains, evaluates predicates,
  returns `(recipeId, bindings)` for `proposeAction`.
- Server verifier:
  [`spacetime/server/modules/shard/src/actions.rs`](../../../spacetime/server/modules/shard/src/actions.rs)
  `propose_action`. Stage 1 walks the tape with `bindings`,
  Stage 2/3/4 handle existence + chain-stitch + locks.
